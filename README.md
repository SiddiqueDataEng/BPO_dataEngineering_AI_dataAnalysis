## 1. Architecture Overview

The platform implements a **Federated Data Mesh** on **Microsoft Azure**, with domain-owned pipelines flowing through a **Medallion (Bronze → Silver → Gold)** lakehouse pattern.

```
┌─────────────────────────────────────────────────────────────────┐
│                      DATA SOURCES                               │
│  CSV / JSON Files   Azure Event Hubs   Call Recordings (ADLS)  │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                    BRONZE LAYER (Raw)                           │
│  • Schema validation    • DLQ routing    • Partitioned parquet  │
│  • Immutable append-only storage         • Event metadata tags  │
└────────────────────────────┬────────────────────────────────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼                  ▼
   ┌─ Insurance ─┐  ┌─ Healthcare ─┐  ┌─ RealEstate ─┐  ┌─ AR ─┐
   │ Domain      │  │ Domain       │  │ Domain        │  │      │
   │ Pipeline    │  │ Pipeline     │  │ Pipeline      │  │      │
   └──────┬──────┘  └──────┬───────┘  └──────┬────────┘  └──┬───┘
          └──────────────────┴──────────────────┴─────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                    SILVER LAYER (Cleaned)                       │
│  • Type casting     • Join enrichment    • State machine        │
│  • PII handling     • Referential checks • Rolling metrics      │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                    GOLD LAYER (Aggregated)                      │
│  • Campaign KPIs    • Agent performance  • Vertical KPIs        │
│  • ML feature store • Compliance summary • Dialer performance   │
└────────────────────────────┬────────────────────────────────────┘
                             │
                    ┌────────┴────────┐
                    ▼                 ▼
             Streamlit            Azure ML
             Dashboard            Training
```

### Technology Mapping

| Layer | Tool | Purpose |
|---|---|---|
| Ingestion | Python pandas / `azure-eventhub` SDK | CSV/JSON/JSONL → Bronze parquet |
| Orchestration | Azure Data Factory + dbt Core | Domain pipeline execution |

| Governance | Microsoft Purview | Catalog, lineage, PII tagging |
| Quality | Great Expectations | Per-domain data quality suites |
| ML Platform | Azure Machine Learning | Feature store, model training |

## 3. Medallion Architecture

The platform uses the **Medallion (Delta Lake) pattern** — three layers with increasing data quality and aggregation.

### Zone Definitions

| Zone | Alias | Mutability | Purpose | Format |
|---|---|---|---|---|
| **Bronze** | Raw | Immutable append-only | Exact copy of source, schema-tagged | Parquet (snappy) |
| **Silver** | Cleaned | Overwrite on reprocess | Validated, typed, enriched, joined | Parquet (snappy) |
| **Gold** | Aggregated | Overwrite on reprocess | Business metrics, KPIs, ML features | Parquet (snappy) |

### Partitioning Strategy

```
lakehouse/
├── bronze/
│   ├── calls/
│   │   ├── partition=ar/calls.parquet
│   │   ├── partition=healthcare/calls.parquet
│   │   ├── partition=insurance/calls.parquet
│   │   └── partition=realestate/calls.parquet
│   ├── leads/
│   │   ├── partition=ar/
│   │   ├── partition=healthcare/
│   │   ├── partition=insurance/
│   │   └── partition=realestate/
│   ├── campaigns/          (partitioned by vertical)
│   ├── ml_features/        (partitioned by vertical)
│   ├── pipeline_events/    (partitioned by payload_vertical)
│   ├── agents/             (not partitioned — small dim table)
│   ├── clients/
│   ├── dnc_list/

├── silver/
```

## 4. Bronze Layer — Raw Ingestion

**File:** `pipeline/bronze/ingest.py`

The Bronze layer is pure **ELT** — extract the source, load it as-is, then let Silver transform. No business logic here.

### What it does

1. Reads each source CSV or JSONL file
2. Validates against the schema contract (`schema_validator.py`)
3. Adds Bronze metadata columns (`_bronze_ingested_at`, `_bronze_source_file`, `_bronze_schema_version`)
4. Writes valid rows to `lakehouse/bronze/{table}/` as Parquet, partitioned by `vertical` where applicable
5. Routes invalid rows to `lakehouse/dlq/` as CSV with `_validation_errors` column

### Running Bronze only

```bash
python pipeline/run_pipeline.py --layer bronze
```

### Bronze metadata columns

Every Bronze table gets three extra columns injected:

| Column | Type | Example |
|---|---|---|
| `_bronze_ingested_at` | ISO datetime | `2026-07-18T11:28:34.123456+00:00` |
| `_bronze_source_file` | string | `output/calls.csv` |
| `_bronze_schema_version` | string | `1.0` |

### Schema validation rules

Required columns are checked per table. For example, `calls` requires:

```python
["call_id", "lead_id", "agent_id", "campaign_id", "vertical",
 "started_at", "ended_at", "duration_seconds", "disposition"]
```

Enum checks are also enforced:

```python
# calls.disposition must be one of:
{"Interested", "NotInterested", "Callback", "DNC",
 "Transfer", "NoAnswer", "Voicemail"}

# campaigns.dialing_mode must be one of:
{"Preview", "Progressive", "Predictive"}
```

### JSONL event flattening

`pipeline_events.jsonl` has a nested `payload` object. The Bronze ingestor flattens it:

```
payload.call_id    →  payload_call_id
payload.vertical   →  payload_vertical
payload.sentiment  →  payload_sentiment_score
```

This makes downstream SQL queries possible without JSON parsing.

### Bronze output sizes (default 1K leads / 5K calls)

| Table | Files | Size |
|---|---|---|
| calls (4 partitions) | 4 | 874 KB |
| pipeline_events (4 partitions) | 4 | 788 KB |
| leads (4 partitions) | 4 | 250 KB |
| qa_reviews | 1 | 132 KB |
| insurance_qualifications | 1 | 57 KB |
| appointments | 1 | 77 KB |
| **Total** | **29** | **~2.5 MB** |

## 5. Silver Layer — Domain Transforms

**File:** `pipeline/silver/transforms.py`

Silver is **ETL** — reads Bronze, applies business rules, writes clean typed tables. Each domain owns its transform function.

### Running Silver only

```bash
python pipeline/run_pipeline.py --layer silver
```

### Transform Functions

#### Platform Dimensions

| Function | Output | Key Operations |
|---|---|---|
| `transform_dim_agents()` | `dim_agents.parquet` | Parse hire_date, compute tenure_days, cast booleans |
| `transform_dim_campaigns()` | `dim_campaigns.parquet` | Parse dates, derive is_active, compute campaign_duration_days |
| `transform_dim_clients()` | `dim_clients.parquet` | Parse created_at |
| `transform_dim_dnc()` | `dim_dnc.parquet` | Deduplicate by phone_e164, parse added_at |

#### Core Tables

**`transform_silver_calls()`** → `silver_calls_all.parquet`

The master calls table, enriched with agent data:

```python
# Derived columns added:
call_date          # YYYY-MM-DD string
call_hour          # 0-23 integer
call_week          # ISO week number
call_month         # 1-12
call_year          # YYYY
is_converted       # True if disposition in (Interested, Transfer)
is_compliance_issue # True if compliance_flagged
talk_minutes       # duration_seconds / 60

# Joined from dim_agents:
agent_role
agent_perf_tier
agent_base_conv_rate
agent_vertical
```

**`transform_silver_leads()`** → `silver_leads.parquet`

Leads with `vertical_data` JSON parsed flat:

```python
# vertical_data JSON unpacked to columns prefixed vd_:
vd_product_type        # Insurance: health/life/auto/home
vd_aca_eligible        # Insurance: bool
vd_specialty           # Healthcare: Cardiology etc.
vd_payer_id            # Healthcare: BCBS001 etc.
vd_interest_type       # RealEstate: buyer/seller
vd_budget_min          # RealEstate: integer USD
vd_original_balance    # AR: float USD
vd_sol_expired         # AR: bool

# Derived:
lead_age_days          # days since created_at
```

#### Domain Tables

| Function | Output | Joins |
|---|---|---|
| `transform_silver_insurance()` | `silver_insurance.parquet` | calls + insurance_qualifications + silver_leads (ins cols) |
| `transform_silver_healthcare()` | `silver_healthcare.parquet` | calls + appointments + silver_leads (hc cols) |
| `transform_silver_realestate()` | `transform_silver_realestate.parquet` | calls + realestate_qualifications + silver_leads (re cols) |
| `transform_silver_ar()` | `silver_ar.parquet` | calls + payment_arrangements + silver_leads (ar cols) |

#### QA & Performance

**`transform_silver_qa()`** → `silver_qa_reviews.parquet`

Unpacks the `scores` JSON column into flat rubric columns:

```python
# scores JSON: {"Opening & Introduction": 4.2, "Script Compliance": 3.8, ...}
# Becomes flat columns:
rubric_opening_introduction   = 4.2
rubric_script_compliance      = 3.8
rubric_product_knowledge      = 3.5
rubric_objection_handling     = 4.0
rubric_compliance_adherence   = 4.5
rubric_closing_technique      = 3.2
rubric_professionalism        = 4.1
```

**`transform_silver_agent_performance()`** → `silver_agent_performance.parquet`

Adds rolling window computation:

```python
# Rolling 3-day below-threshold flag (key for auto-flagging agents):
below_threshold_3d = (
    df.groupby("agent_id")["below_threshold"]
    .transform(lambda x: x.rolling(3, min_periods=3).sum() == 3)
)
```

## 7. Federated Data Mesh

The platform follows **four core Data Mesh principles** adapted for a single engineering team:

### Principle 1 — Domain Ownership

Each vertical owns its pipeline, schema contract, and data quality SLA:

| Domain | Owner Role | Bronze Partition | Silver Table | Gold Table |
|---|---|---|---|---|
| Insurance | Insurance Lead | `partition=insurance` | `silver_insurance` | `gold_insurance_kpis` |
| Healthcare | Healthcare Lead | `partition=healthcare` | `silver_healthcare` | `gold_healthcare_kpis` |
| Real Estate | RE Lead | `partition=realestate` | `silver_realestate` | `gold_realestate_kpis` |
| AR Sales | AR Lead | `partition=ar` | `silver_ar` | `gold_ar_kpis` |
| Platform Ops | Data Eng Lead | (all) | `silver_calls_all` | `gold_campaign_kpis` |

### Principle 2 — Data as a Product

Each domain publishes versioned data products with defined SLAs:

```
Data Product: dp_insurance_calls
  Schema version: 1.0
  Owner: Insurance Domain Lead
  SLA: Available within 30 minutes of call completion
  Quality: 0 null call_ids, conversion_rate in [0, 1]
  Consumers: Reporting Service, Azure ML
```

### Principle 3 — Self-Serve Platform

Shared infrastructure that any domain can use without central team bottleneck:

```
Shared Infrastructure:
├── Azure Event Hubs  (event streaming backbone)
├── ADLS Gen2 / OneLake  (shared storage)
├── Azure Data Factory  (pipeline execution engine)
├── dbt Core  (SQL transformation framework)
├── Microsoft Purview  (governance — cross-domain)
└── DuckDB  (interactive query — no server needed)
```

### Principle 4 — Federated Governance

Microsoft Purview enforces policies **without** centralizing ownership:

- **PII classification**: Auto-tags name, DOB, phone, member_id columns in Healthcare domain
- **HIPAA tagging**: All Healthcare Silver/Gold tables tagged `data_classification=PHI`
- **FDCPA tagging**: All AR Silver/Gold tables tagged `data_classification=financial_pii`
- **Lineage**: Source CSV → Bronze partition → Silver join → Gold aggregate tracked automatically
- **Quality scores**: Great Expectations results published as Purview data quality scores

### Data Mesh vs Centralized — Comparison

| Aspect | Old Centralized | Data Mesh (This Platform) |
|---|---|---|
| ETL ownership | Single data team | Domain team per vertical |
| Schema changes | Central approval needed | Domain can evolve independently |
| Pipeline failures | One failure blocks all | Insurance failure doesn't affect AR |
| Scaling | All verticals scale together | Each domain scales independently |
| Quality accountability | Central data team | Domain team accountable |
| Governance | Central catalog | Federated Purview policies |

## 8. Schema Contracts & Validation

**File:** `pipeline/utils/schema_validator.py`

Schema contracts define the minimum required columns and allowed enum values per table. Validation runs at Bronze ingestion and returns a `(valid_df, invalid_df)` tuple.

### Required Columns Per Table

```python
SCHEMAS = {
    "leads": [
        "lead_id", "campaign_id", "vertical", "first_name", "last_name",
        "phone_e164", "state", "status", "dnc_flagged", "created_at"
    ],
    "calls": [
        "call_id", "lead_id", "agent_id", "campaign_id", "vertical",
        "started_at", "ended_at", "duration_seconds", "disposition"
    ],
    "campaigns": [
        "campaign_id", "name", "vertical", "client_id",
        "dialing_mode", "status", "created_at"
    ],
    "appointments": [
        "appointment_id", "call_id", "lead_id", "patient_name",
        "appointment_date", "provider_name", "facility",
        "payer_id", "member_id"
    ],
    # ... (see schema_validator.py for full list)
}
```

### Enum Validation

```python
ENUM_CHECKS = {
    "leads": {
        "vertical":  {"Insurance", "Healthcare", "RealEstate", "AR"},
        "status":    {"New", "Dialing", "Contacted", "Dispositioned",
                      "Recycled", "Closed"},
    },
    "calls": {
        "disposition": {"Interested", "NotInterested", "Callback",
                        "DNC", "Transfer", "NoAnswer", "Voicemail"},
    },
    "campaigns": {
        "dialing_mode": {"Preview", "Progressive", "Predictive"},
        "status":       {"Draft", "Active", "Paused", "Closed"},
    },
    "agents": {
        "performance_tier": {"high", "mid", "low"},
        "role": {"Agent", "Team Lead", "Supervisor", "QA Manager",
                 "Campaign Manager", "Data Engineer", "Data Scientist", "Admin"},
    },
}
```

### Validation Flow

```
Input DataFrame
      │
      ├── Check for missing required columns
      │     └── If any missing → entire df goes to invalid
      │
      ├── Check for null values in required columns
      │     └── Rows with any null → invalid_df
      │
      ├── Check enum values
      │     └── Rows with invalid enum values → invalid_df
      │
      └── valid_df → Bronze parquet
          invalid_df → DLQ CSV (with _validation_errors column)
```

### Example DLQ Record

```csv
qualification_id,call_id,lead_id,product_type,...,_validation_errors
abc123,xyz789,def456,,,,"null:date_of_birth; "
```

## 9. Dead Letter Queue (DLQ)

**Location:** `lakehouse/dlq/`

The DLQ captures rows that fail schema validation during Bronze ingestion. These are stored as timestamped CSV files for inspection and reprocessing.

### DLQ File Naming

```
{table}_dlq_{YYYYMMDD_HHMMSS}.csv
```

Example:
```
insurance_qualifications_dlq_20260718_110555.csv
```

### DLQ Record Structure

Every DLQ record includes all original columns plus:
- `_validation_errors` — semicolon-separated list of failure reasons

## 11. Data Models — Full Schema Reference

### Lead State Machine

```
New ──→ Dialing ──→ Contacted ──→ Dispositioned ──→ Recycled
                                        └──────────────→ Closed

Invalid transitions (return HTTP 422):
  - Closed → any state
  - New → Dispositioned (must go through Dialing/Contacted first)
  - Recycled → Closed (must re-enter via New)
```

### Call Disposition Taxonomy

```
Dispositions
├── Positive (converted = True)
│   ├── Interested      — lead expressed interest
│   └── Transfer        — warm-transferred to closer/broker
│
├── Pending (recyclable)
│   ├── Callback        — scheduled callback requested
│   ├── NoAnswer        — no answer (auto-recyclable after N days)
│   └── Voicemail       — voicemail left
│
└── Terminal
    ├── NotInterested   — explicit rejection
    └── DNC             — added to do-not-call list
```

### E.164 Phone Number Format

```
│ │  │   └── Subscriber number (7 digits)
│ │  └────── Area code (3 digits)
│ └───────── Country code (1 = USA)
└─────────── + prefix (required)

Valid range: +1 followed by 10 digits for US numbers
```

### Query Examples by Complexity

#### Basic — Campaign KPIs
```sql
SELECT
    name                                    AS campaign,
    vertical,
    total_calls,
    ROUND(conversion_rate * 100, 2)         AS conv_rate_pct,
    compliance_flags
FROM gold_campaign_kpis
ORDER BY conv_rate_pct DESC;
```

#### Intermediate — Healthcare Funnel
```sql
SELECT
    call_month,
    SUM(total_calls)                                AS calls,
    SUM(total_appointments_scheduled)               AS appointments,
    ROUND(AVG(appointment_rate) * 100, 2)           AS appt_rate_pct,
    SUM(webhook_sent_count)                         AS webhooks_sent
FROM gold_healthcare_kpis
GROUP BY call_month
ORDER BY call_month;
```

#### Window Function — Running Conversion Rate
```sql
SELECT
    campaign_id,
    call_month,
    total_calls,
    SUM(total_calls)
        OVER (PARTITION BY campaign_id ORDER BY call_month
              ROWS UNBOUNDED PRECEDING)             AS running_calls,
    ROUND(
        SUM(total_converted)
            OVER (PARTITION BY campaign_id ORDER BY call_month
                  ROWS UNBOUNDED PRECEDING) * 100.0
        / NULLIF(SUM(total_calls)
            OVER (PARTITION BY campaign_id ORDER BY call_month
                  ROWS UNBOUNDED PRECEDING), 0)
    , 2)                                            AS running_conv_pct
FROM gold_insurance_kpis
ORDER BY campaign_id, call_month;
```

#### Window Function — Month-over-Month Change (LAG)
```sql
WITH monthly AS (
    SELECT
        vertical,
        STRFTIME(CAST(call_date AS DATE), '%Y-%m') AS month,
        SUM(total_dials)                            AS dials
    FROM gold_dialer_performance
    GROUP BY vertical, STRFTIME(CAST(call_date AS DATE), '%Y-%m')
)
SELECT
    vertical, month, dials,
    LAG(dials) OVER (PARTITION BY vertical ORDER BY month) AS prev_dials,
    ROUND(
        (dials - LAG(dials) OVER (PARTITION BY vertical ORDER BY month))
        * 100.0
        / NULLIF(LAG(dials) OVER (PARTITION BY vertical ORDER BY month), 0)
    , 2)                                                    AS mom_change_pct
FROM monthly
ORDER BY vertical, month;
```

#### CTE — Compliance Risk Score
```sql
WITH compliance_monthly AS (
    SELECT vertical, call_month,
        ROUND(AVG(compliance_flag_rate) * 100, 4) AS flag_rate_pct
    FROM gold_compliance_summary
    GROUP BY vertical, call_month
),
risk_score AS (
    SELECT *,
        ROUND(flag_rate_pct * 0.35, 4) AS weighted_score
    FROM compliance_monthly
)
SELECT *,
    CASE
        WHEN weighted_score > 3.5 THEN 'HIGH'
        WHEN weighted_score > 1.5 THEN 'MEDIUM'
        ELSE 'LOW'
    END AS risk_level
FROM risk_score
ORDER BY weighted_score DESC;
```

#### Advanced — ML Lift Analysis
```sql
WITH scored AS (
    SELECT lead_id, vertical, converted,
        COALESCE(predicted_score, global_predicted_score) AS score
    FROM gold_ml_feature_store
    WHERE COALESCE(predicted_score, global_predicted_score) IS NOT NULL
),
deciled AS (
    SELECT *, NTILE(10) OVER (ORDER BY score DESC) AS decile
    FROM scored
),
baseline AS (
    SELECT ROUND(AVG(CAST(converted AS DOUBLE)) * 100, 4) AS base_rate
    FROM scored
)
SELECT
    d.decile,
    COUNT(*)                                        AS leads,
    ROUND(AVG(d.score), 1)                          AS avg_score,
    ROUND(AVG(CAST(d.converted AS DOUBLE))*100, 2)  AS conv_rate_pct,
    ROUND(AVG(CAST(d.converted AS DOUBLE))*100
          / b.base_rate, 2)                         AS lift
FROM deciled d CROSS JOIN baseline b
GROUP BY d.decile, b.base_rate
ORDER BY d.decile;
```

### Important DuckDB Notes

> `call_date` is stored as VARCHAR (string). Always cast explicitly:
> ```sql
> STRFTIME(CAST(call_date AS DATE), '%Y-%m')
> ```

> For `COALESCE` with numeric columns, avoid mixing with string `'-'`:
> ```sql
> -- WRONG (type error):
> COALESCE(numeric_col, '-')
> -- CORRECT:
> COALESCE(numeric_col, NULL)
> ```

## Appendix — Key Formulas

| Metric | Formula |
|---|---|
| Conversion Rate | `(Interested + Transfer) / Total Calls` |
| Abandon Rate | `Abandoned / Total Dials` |
| Answer Rate | `Answered / Total Dials` |
| ACA Eligibility Rate | `ACA Eligible Count / Total Calls` |
| Recovery Rate (AR) | `Settlement Amount / Original Balance` |
| Appointment Rate | `Appointments Scheduled / Total Calls` |
| Qualification Rate | `Qualified Leads / Total Calls` |
| DNC Rate | `DNC Dispositions / Total Calls` |
| Compliance Flag Rate | `Flagged Calls / Total Calls` |
| Agent Lift | `Agent Conv Rate / Campaign Avg Conv Rate` |
| Model Lift (Decile) | `Decile Conv Rate / Overall Conv Rate` |
| AUC-ROC | Area under TPR vs FPR curve — range [0.5, 1.0] |
| Sentiment Score | NLP float — range [-1.0 (negative), +1.0 (positive)] |
| Lead Score | ML propensity — range [0 (unlikely), 100 (highly likely)] |
